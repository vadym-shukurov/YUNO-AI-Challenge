# Yuno · Payment Orchestration Dashboard

A prototype operations dashboard for an Indonesian payment orchestrator. It monitors processor health, flags approval-rate anomalies, and lets operators drill into individual transactions.

## Getting started

```bash
npm install
npm run dev
```

Open <http://localhost:3000>. Filters sync to the URL (`?range=14d&method=GoPay`), so dashboard views are shareable.

Docker (one-liner):

```bash
docker build -t yuno-dashboard . && docker run --rm -p 3000:3000 yuno-dashboard
```

Other scripts:

- `npm run build` — production build
- `npm run start` — run the production build
- `npm run typecheck` — strict TypeScript check
- `npm run lint` — Next/ESLint
- `npm run test` — unit/integration tests (Vitest)
- `npm run test:coverage` — coverage run (thresholds enforced)
- `npm run test:e2e` — Playwright E2E (starts production server automatically)
- `npm run test:load` — Artillery smoke load test (starts server automatically)
- `npm run test:a11y` — Playwright + axe-core accessibility scan (`e2e/a11y.spec.ts`)
- `npm run test:lighthouse` — Lighthouse CI (optional; needs arm64 Node on Apple Silicon; starts server automatically)
- `npm run test:chaos` — Playwright chaos tests (latency + param-fuzz; starts server automatically)
- `npm run test:security` — lightweight security checks + dependency audit (starts server automatically)
- `npm run test:all` — Vitest + full Playwright E2E (including axe) + Artillery + security checks

## Reliability, recovery, monitoring

- **Error handling**: the App Router error boundaries (`src/app/error.tsx`, `src/app/global-error.tsx`) provide a user-facing recovery path and emit structured logs.
- **Recovery**: `src/lib/recovery/retry.ts` provides exponential backoff with jitter to safely retry transient operations without thundering-herd behavior.
- **Monitoring (SLA/SLO-ready)**:
  - Health endpoint: `GET /api/health` (for uptime checks)
  - CSP report sink: `POST /api/csp-report` (bounded payload; returns 413 for oversized reports)
  - Structured logs: `src/lib/observability/logger.ts` outputs JSON events that are easy to ingest in most log pipelines.

Suggested SLOs for a dashboard like this:

- **Availability**: 99.9% monthly (based on `GET /api/health`)
- **Latency**: p95 < 500ms page-load TTFB for `/` (production), p95 < 100ms for `GET /api/health`
- **Errors**: < 0.1% of sessions hitting `app.*error_boundary` events

## Mock data & smart-routing logic

The dashboard runs against 600 deterministic mock transactions generated by [`src/lib/generateMockData.ts`](src/lib/generateMockData.ts). The generator encodes the routing behavior we want operators to observe:

| Rule | Implementation |
| --- | --- |
| **Processor weights** | A 40% · B 25% · C 20% · D 10% · E 5%. Draw is weighted-random unless a method override fires first. |
| **Method pinning** | 70% of GoPay volume is pinned to Processor B (an Indonesia-specific co-brand agreement). Remaining GoPay and all other methods fall back to the weighted draw. |
| **Overall status mix** | 82% Approved / 15% Declined / 3% Failed. |
| **Processor B anomaly** | For the last 48 hours, Processor B's approval rate collapses to 65% (vs. 88% baseline). The 35% of failed authorizations is split between Declined and Failed in proportion to the overall decline-vs-fail ratio, so the shape of the failure tail is realistic. |
| **Baseline rates (`PROCESSOR_METRICS`)** | A 90% · B 88% · C 85% · D 80% · E 75%. Exposed as a constant so anomaly detection can compare “current window” against “steady state” without re-training. |

Amounts are drawn uniformly from 50k–5M IDR, timestamps are uniformly distributed across the last 14 days, and the array is returned newest-first.

## Architecture

```
src/
├── app/                    # Next.js 14 App Router (layout, page, providers)
├── components/
│   ├── dashboard/          # KPIs, charts, smart alerts, overview composition
│   ├── filters/            # Global date + payment-method filter widgets
│   ├── layout/             # Sidebar, header, responsive grid
│   ├── transactions/       # TanStack table + detail drawer
│   └── ui/                 # Shadcn-style primitives
└── lib/
    ├── constants/          # Processor colors, shared lookups
    ├── filters/            # nuqs-backed filter hook (URL state)
    ├── hooks/              # useProcessedData — filter + aggregate in one place
    ├── transactions/       # Deterministic routing-detail derivation
    ├── generateMockData.ts # Mock payment data generator
    └── utils.ts            # cn(), currency / percent formatting
```

Global filter state lives in the URL (`range`, `from`, `to`, `method`, `processor`, `outcome`, `min`, `max`, `q`, …) via [`nuqs`](https://nuqs.47ng.com/). Every widget — charts, table, alerts — reads filters through the same [`useProcessedData`](src/lib/hooks/use-processed-data.ts) hook, which filters the mock dataset and memoizes all aggregates.

**Smart Alerts** evaluate the **trailing 48 hours** ending at the selected range’s `to` timestamp (not the entire 14‑day blend). A processor is flagged only when (a) it has at least **`MIN_ALERT_VOLUME` (15)** transactions in that 48h slice and (b) the approval-rate drop exceeds `max(10pp, ~2σ)` for that baseline/volume — this mirrors how ops teams avoid noisy “single-digit sample” pages.

**Trade-off (intentional):** the **Approval Rate Over Time** line chart buckets the **entire** selected date range, while **Smart Alerts** and the **Approval vs Baseline** “Current” bars emphasize the **trailing 48h** slice. That pairing is common in ops tooling: the line answers “how did we trend across the investigation window?”, while the bars + alerts answer “what is happening *right now* at the tail?”.

## Technical choices

- **Shadcn UI pattern (speed).** Copy-in, owned primitives mean we can ship a cohesive fintech look without a heavy component library. Tailwind tokens (`--primary`, `--success`, etc.) are the single source of truth for theming, and every component is readable in one file if an engineer needs to audit accessibility or swap a color.
- **Recharts (flexibility).** The anomaly story needs three very different visualizations (donut, grouped bars with conditional coloring, time-series line with a reference band). Recharts composes from primitive elements, so coloring a single bar red when `baseline − current > 10%`, or drawing a dashed 82% target line, is a one-liner. SVG output also plays well with server components.
- **TanStack Table (performance).** The Transaction Explorer is headless — TanStack handles 600 rows today and will handle 60k the moment the prototype hits a real API, because sort / filter / paginate run on memoized row models rather than full re-renders. Decoupling state from markup also lets us swap in virtualization later without rewriting columns.

## Testing & coverage

Tests run with **Vitest** + Testing Library. Coverage is enforced at **≥85%** on `src/lib/**` (business logic, filtering, aggregation, and routing details). UI primitives and App Router files are intentionally excluded from the threshold to keep the signal focused on the logic that drives dashboard correctness.

### E2E testing (Playwright)

The E2E suite targets end-user journeys:

- Dashboard loads (smoke)
- URL-synced filters (method/processor/outcome/amount/date)
- Transaction drill-down (row click → details)
- Anomaly surfacing on recent windows
- Chaos scenarios (synthetic latency + param-fuzz)

We use stable selectors (`data-testid`) for maintainability.

Run:

```bash
npm run test:e2e
```

### Load testing (Artillery)

Artillery runs a small “production smoke” load scenario against:

- `GET /`
- `GET /api/health`
- `GET /` with a representative filter URL

Run:

```bash
npm run test:load
```

### Accessibility testing

**Default (`npm run test:a11y`):** Playwright + **axe-core** (`e2e/a11y.spec.ts`). This runs in CI and on developer machines regardless of CPU/Node architecture and catches serious/critical WCAG issues on the real hydrated UI.

**Optional Lighthouse (`npm run test:lighthouse`):** Lighthouse CI reads [`.lighthouserc.json`](.lighthouserc.json), starts the app via [`scripts/with-server.mjs`](scripts/with-server.mjs), and writes HTML reports under `./lighthouse-report/`.

On **macOS Apple Silicon**, Lighthouse’s bundled Chrome path expects **arm64 Node**. If `node -p process.arch` prints `x64` on an `arm64` machine, Lighthouse will exit with an explicit Rosetta warning — use an arm64 Node build (e.g. native Homebrew Node) or rely on `npm run test:a11y` for accessibility gates.

```bash
npm run test:a11y
npm run test:lighthouse
```

### Security testing

`npm run test:security` performs:

- basic endpoint checks (`/`, `/api/health`, `/api/csp-report`)
- a reflected-XSS guard on the HTML response
- `npm audit --omit=dev`

This is **not** a replacement for a full SAST/DAST program, but it is a practical baseline for a prototype.

## Insight summary

**GoPay routed through Processor B broke in the last 48 hours.** On the 14-day view the overall approval rate sits around the 82% target, but narrow the date range to the last 24–48 hours and Processor B's approval rate drops to ~65% — a 23-point swing from its 88% baseline. The Smart Alerts strip surfaces this as a *Critical* card the moment the window is tightened. Because of the 70% GoPay → Processor B pin, the failure concentrates in a single method: filter the global picker to **GoPay** and the Approval Trend line caves in over the final two days while every other method continues to perform within tolerance. This is the "why" behind the aggregate regression that is easy to miss when only looking at a blended number.

**The Processor Mix donut shows we were over-exposed.** Processor B is the second-largest node by design (~25% of total volume), but once the GoPay pin is active its effective share of *GoPay* volume balloons past 70%. The donut makes the concentration visible: one underperforming node is handling a large, concentrated slice of a single payment method, so a localized issue propagates directly to top-line approval rate. Pairing the donut with the "Approval Rate vs Baseline" bar chart (where Processor B turns red against a muted-gray baseline) closes the loop — we can see *how much* traffic was exposed and *how far below baseline* that traffic is landing, in one glance.

**Recommended action: temporarily re-route GoPay off Processor B.** Update the smart-routing rule for `paymentMethod = GoPay` so traffic prefers **Processor A** (baseline 90%, currently healthy) until Processor B's 60-minute rolling approval rate recovers to within 5 points of its 88% baseline for two consecutive hours. Keep Processor B eligible for non-GoPay traffic so we preserve commercial-cost advantages where it still clears, and alert the processor's account manager with the decline/failure breakdown from the Transaction Explorer (`R01 · Issuer risk decline` and `TIMEOUT` dominate the failed cohort) so their team has a concrete lead. Re-evaluate after the next reconciliation window; if the regression persists, demote Processor B's overall weight from 25% to 10% until root-cause is confirmed.
